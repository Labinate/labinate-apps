How to put GitHub on your resume

A GitHub link is one of the few things on a resume a recruiter can click and verify in ten seconds — which is exactly why an empty or messy profile does more harm than no link at all. This isn't about where the URL goes; it's about making the profile earn the click: which repos to pin, what a reviewer actually looks at, and how to reference a specific project in the resume itself.

Updated July 2026 · ~6 min read · by Labinate

Most resume advice treats GitHub as one more link to drop in the header. For an engineer it's more than that: it's the only line on the page a hiring manager can independently confirm. A strong profile turns "I can build" from a claim into evidence. A weak one turns the same link into evidence against you. So the real question isn't whether to add the URL — it's whether your profile is in a state worth opening, and how to point the reader at the two or three things that make your case. This guide covers the decision, the curation, and how to reference a repo inside the resume body.

Should you include GitHub at all?

For most engineering roles, yes — once the profile is worth clicking. The link is an asset or a liability depending entirely on what's behind it.

Include it → when you have real, presentable projects

Three to six pinned repos you actually built, each with a README that explains what it does — this is a genuine advantage, especially for developers, data, and infrastructure roles. It's proof that survives the interview.

Curate first → when the profile is a pile of forks and half-starts

A profile that's mostly forks, tutorial follow-alongs, and empty repos reads as "nothing to show." Spend an hour pinning and cleaning up (steps below) before you add the link — or leave it off until it's ready.

Skip it → when your best work can't be public

If everything you've built is proprietary or under NDA, don't link a thin profile just to have the line. Lean on your project bullets and a portfolio instead. A GitHub link is optional for non-engineering roles — include it only if code is part of the job.

What a reviewer actually looks at

When a technical reviewer opens your GitHub, they don't scroll your whole history. In the first fifteen seconds they check, in order:

  • Your pinned repositories. This is the shelf you chose to display, so it's judged as a curated portfolio — not "here's everything," but "here's my best."
  • Whether the top repos have READMEs. A README that says what the project is, why it exists, and how to run it signals someone who finishes things. A repo with no README reads as abandoned.
  • Recency and your own commits. Do the commits look like real, incremental work, or a single "initial commit" dump? A project with a commit history tells a story; a one-shot upload doesn't.
The contribution-graph myth. The green streak grid is the thing beginners obsess over and reviewers largely ignore. It's trivial to game with empty commits, so a dense graph proves almost nothing — and manufacturing one is effort spent where no one's looking. Put that hour into making three repos presentable instead. That's the screen a reviewer actually clicks into.

Curate before you link (about an hour)

You don't need more projects — you need the ones you have to look intentional. Do this before the link goes on the resume:

  1. Pin your three to six strongest repos. On your profile, use "Customize your pins" and choose projects relevant to the role, ones you can explain end to end. Quality of the shelf beats quantity of repos.
  2. Write a real README for each pinned repo. One paragraph on what it does and why, a short "how to run it," and a line on the stack. This is the single highest-leverage hour you can spend on your profile.
  3. Bury the noise. You can't delete forks in bulk, but you don't have to feature them — pinning controls what shows first. Consider archiving dead repos so they read as "finished," not "forgotten."
  4. Set a profile README and a clean handle. A short profile README (a repo named exactly your username) gives a one-line intro and links to your best work. A professional username beats xX_devgod_Xx on a resume.

How to reference a repo in the resume body

The header link says "here's my GitHub." Sometimes one project is central enough to your case that it deserves a line in the resume itself — in a projects section or an experience bullet. Do it the way you'd write any strong bullet: lead with what it does and the impact, name the stack, and let the short link sit at the end.

Weak — just a link Built a project on GitHub: github.com/janedoe/parser
Strong — outcome, stack, then the link Log-parsing CLI (Go) — open-source tool that cut a manual triage step from ~30 min to under 1 min; 400+ stars, used in 3 companies' CI. github.com/janedoe/logparse

A few rules that keep this from backfiring:

  • Only link a repo you'd demo live. A bullet that points at a specific repo invites a click into that exact repo — make sure it's your strongest, with a clean README.
  • Quantify like any other bullet. Stars, downloads, users, or the time/effort it saved. "A tool I made" is a hobby; "a tool 3 teams use" is an achievement.
  • Don't repeat the header. If a repo isn't worth a full bullet, leave it to the profile link. One or two named projects, not a URL after every line.

Where the link goes and how to format it

The profile URL lives in your contact header at the top, alongside your email, phone, and LinkedIn — never in its own section. Format it the same clean way as every other link:

Contact header — links formatted consistently Jane Doe · Austin, TX · jane.doe@email.com · linkedin.com/in/janedoe · github.com/janedoe

Write it as plain readable text (github.com/janedoe, no https://www.) and make it a live hyperlink so a PDF stays one click while an ATS still parses the text. For the full mechanics of link placement and formatting across portfolio, GitHub, and personal sites, see how to list a portfolio link.

When a GitHub link hurts you

  • A profile that's only forks and course exercises. An empty or copycat GitHub reads worse than none — it answers "what have you built?" with "nothing of my own."
  • Pinned repos with no README. A reviewer who clicks in and can't tell what the project does moves on, and the whole link deflates.
  • An unprofessional username on a formal resume. The handle is part of the link; make it your name or something clean.
  • A stale account you're padding to look active. Manufactured commits and a link to abandoned work signal the opposite of what you intend. If it's not ready, ship the resume without it.
The whole thing in one breath. Add GitHub when the profile is worth opening — pin three to six real projects, give each a README, and ignore the contribution graph. Put the clean link (github.com/you, live hyperlink) in your contact header with LinkedIn and email, and if one project is central, give it a quantified bullet in the body. Never link a profile you wouldn't walk a hiring manager through line by line.

Frequently asked questions

Should I put GitHub on my resume?

Only if the profile is worth opening. For most engineering roles a link with real, pinned projects is a strong asset — live proof you can build. But a profile of forks, tutorial follow-alongs, and empty repos hurts more than no link, because a recruiter reads it as evidence you have nothing to show. Curate it first — pin three to six real projects with readable READMEs — then add the link.

Which repositories should I pin on my GitHub?

Three to six repos you built and can talk through in an interview. Favor projects relevant to the role, with a README explaining what they do and how to run them, and your own commit history rather than a one-time upload. A small finished tool with a clean README beats a large half-abandoned repo. Skip forks and course exercises.

Does the green contribution graph matter to recruiters?

Far less than people think. It's easy to game with trivial commits, so experienced reviewers largely ignore it and look at the actual repositories — what you built and whether the code and READMEs read well. Don't manufacture streaks; spend the effort making three real projects presentable.

Where does the GitHub link go on a resume?

In your contact header at the top, next to your email, phone, and LinkedIn, formatted as clean text like github.com/yourname and made a live hyperlink in the PDF. If one project is central to your case, you can also reference that single repo directly in the relevant projects or experience bullet.
